Shadow — Page Architecture & SEO Map
1. Purpose of this document

This document defines the site structure, landing architecture, navigation, conversion flow, SEO direction and page system for Shadow. The landing page should not exist as an isolated art piece. It should be part of a coherent product presence that can support portfolio review, early access, future content and trust.

2. Recommended site structure
Primary pages

/ — main cinematic landing page

/manifesto — origin story, philosophy and product thesis

/how-it-works — product mechanism, memory loop, AI architecture explained simply

/labs — self-discovery layer and experimental experiences

/privacy — privacy, memory control and data position

/about — founder/team note and project context

/early-access — waitlist/signup page

Optional future pages

/blog — essays on second brain, personal AI, cognitive overload, life analytics and AI memory

/changelog — product development updates

/demo — interactive narrated walkthrough

/case-study — portfolio-oriented build breakdown

/press-kit — logo, Shady, screenshots and short description

3. Main landing page architecture
Section 1 — Hero / Awakening

Goal: introduce the big promise and Shady.

Content: headline, subheadline, CTA, Shady awakening, raw signal fragments.

Primary CTA: Join early access / Enter Shadow.

Secondary CTA: See how it works.

Section 2 — Problem / Chaos Field

Goal: make the user feel the pain of fragmented life signals.

Content: thoughts, tasks, emotions, goals, habits, money, events, relationships scattered around Shady.

Proof: recognizable raw input examples.

Section 3 — Capture / Structure

Goal: show the core product mechanism.

Content: Inbox Dump, AI Classification, Clarifying Question, structured output.

Proof: raw input → structured memory.

Section 4 — Living Memory

Goal: show continuity and differentiation from notes/chatbots.

Content: Memory Graph, connected nodes, patterns over time.

Proof: old signals connecting to new ones.

Section 5 — Modes

Goal: show product depth and state-based UX.

Content: Capture, Focus, Reflection, Planning, Recovery, Deep Analysis, Labs.

Proof: Shady transforms with each mode.

Section 6 — Life Circle / Analytics

Goal: show personal analytics and life map.

Content: Life Circle, state sliders, energy, habits, emotions, goals, recent signals.

Proof: daily signals update analytics.

Section 7 — Inner Dialogue

Goal: show AI questions and thinking support.

Content: raw input, clarifying question, answer, saved memory.

Proof: specific question that feels useful.

Section 8 — Labs

Goal: show mystery, depth and self-discovery.

Content: Labs cards, archetypes, scans, results saved to memory.

Proof: sample result output.

Section 9 — Trust / Proof

Goal: reduce skepticism and privacy concerns.

Content: privacy position, founder note, product proof, roadmap.

Proof: real outputs and user control language.

Section 10 — Final Portal / CTA

Goal: conversion.

Content: Shady opens final portal, all system elements converge.

CTA: Join early access.

Secondary link: Read manifesto / Privacy.

12. Technical SEO requirements

Use server-rendered metadata.

Add unique title and meta description per page.

Add Open Graph and Twitter card metadata.

Use semantic headings despite cinematic layout.

Use descriptive alt text for product visuals.

Keep heavy animations lazy-loaded.

Use reduced-motion support.

Keep initial hero performant.

Avoid hiding all meaningful text inside canvas/WebGL.

Ensure mobile performance is strong.

Add sitemap.xml and robots.txt.

Add structured data for software application when product details are stable.

13. Architecture risks

If everything lives on one cinematic page, trust and SEO will be weak.

If there are too many pages too early, the product may feel overbuilt.

If the landing has strong visuals but no clear product mechanism, users will not convert.

If SEO copy is too conventional, Shadow will lose its premium atmosphere.

The recommendation is to launch with a strong homepage plus /manifesto, /how-it-works, /privacy and /early-access. Labs and case study can follow.
